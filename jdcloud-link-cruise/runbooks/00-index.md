## 通用执行约定

### 安全铁律（Safety Gates）

以下规则适用于所有场景，**不可覆盖、不可跳过**：

| 操作类型 | 安全要求 | 违规后果 |
|---|---|---|
| **任何资源的删除/释放**（VM/CLB/EIP/Redis/RDS/NAT/K8s） | ❌ 不允许通过巡检脚本执行，必须用户明确确认后由对应 ops skill 操作 | Safety = 0，GCL 立即 ABORT |
| **任何资源的停止/关机**（VM/Redis/RDS） | ❌ 不允许自动执行，必须用户明确确认 | Safety = 0，GCL 立即 ABORT |
| **任何资源的规格变更/降配** | ❌ 不允许自动执行，巡检只出具建议 | Safety = 0，GCL 立即 ABORT |
| **巡检脚本本身代码变更**（增删改 Analyzer） | ✅ 执行前需人工 Review + GCL 门 | — |
| **巡检脚本配置变更**（阈值调整） | ✅ 无需确认，但需记录 changelog | — |

> **底线：本 Skill 是纯读（Read-Only）巡检，不执行任何写操作。**
> 任何需要变更资源的结论，报告只输出"建议"，具体操作由人工通过对应 ops skill 执行。

### 执行流程约定

所有场景统一遵循 **两阶段执行模式**：

```

```
Phase 1: 嗅探 + 拓扑初判
  → 自动发现资源 → 分类部署模式 → 输出拓扑初判报告
  → 置信度 > 0.8 自动继续；≤ 0.8 等待人工确认

Phase 2: 深度巡检
  → 按确认后的拓扑分派 Analyzer
  → 采集监控 + 环比 + 告警历史
  → 逐资源分析 + 链路关联推理 + 规格上限检查
  → 输出双格式报告（Markdown + JSON）
```

## 验收标准定义

| 等级 | 定义 | 阈值规则 |
|---|---|---|
| **PASS** | 全部指标正常，无异常趋势，无告警 | 所有维度评分 ≥ 0.5 |
| **WARNING** | 存在指标超出警告阈值，或有上升趋势 | 任一维度评分 < 0.5 且 ≥ 0.3 |
| **CRITICAL** | 存在指标超出严重阈值，或服务不可用 | 任一维度评分 < 0.3，或 Safety = 0 |
| **ERROR** | 巡检流程本身执行失败（API 不可达等） | 脚本异常退出 |

> 详细评分矩阵见 `references/severity-matrix.md`

## 持续改进闭环

```
┌────────────────────────────────────────────────────────┐
│                   改进闭环流程                           │
│                                                        │
│  巡检执行 ──▶ 报告输出 ──▶ 人工审阅 ──▶ 问题录入       │
│     │                      │             │              │

│     │                 ┌──────────┐ ┌──────────┐        │
│     │                 │ 阈值调优  │ │ 新增场景  │        │
│     │                 └────┬─────┘ └────┬─────┘        │
│     │                      │             │              │

│                          │                              │
│                          ▼                              │
│                  runbook 版本更新                        │
│                   （SemVer + Changelog）                 │
│                                                        │
│  版本: 1.0.0 │ 最后更新: 2026-06-06 │ 作者: pi-agent  │
└────────────────────────────────────────────────────────┘
```

改进触发条件：
1. **误报/漏报** — 人工审阅发现评分与实际情况不符 → 调阈值
2. **新发现** — 巡检中发现未覆盖的链路组件 → 新增 Analyzer + 更新 runbook
3. **架构变更** — 客户业务架构变化（如新上 K8s）→ 更新 runbook 流程
4. **API 变更** — JD Cloud API 版本更新导致调用失败 → 更新 `jdc_client.py`
