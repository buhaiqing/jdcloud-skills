# Agent Skill 评审体系（极简生产版）

## 目标

用最少流程确保 Agent Skill 在生产可用、可控、可持续演进。

## 范围边界（避免职责重叠）

- 本文档聚焦：**Skill 的工程化发布治理**（关卡、评分、发布决策、组合审查、生命周期）
- Agent/RAG 应用质量细则（如检索质量与模型可靠性）由 `10_ai_agent_rag_guide.md` 负责
- 非确定性系统评估统计方法由 `14_nondeterministic_eval_guide.md` 负责
- 对抗性/红队审查方法由 `15_adversarial_review_guide.md` 负责
- Skill 全生命周期审查由 `16_skill_lifecycle_review.md` 负责

## 一、评审关卡（5关）

1. **Design Gate**：目标、输入输出、边界、失败策略是否清晰
2. **Security Gate**：权限、数据、注入、防越权是否达标
3. **Composition Gate**：Skill 组合调用是否安全、权限是否提升、上下文是否泄漏
4. **Reliability Gate**：超时、重试、降级、幂等是否完备
5. **Release Gate**：评估结果、监控指标、回滚方案是否齐备

## 二、阻断规则（任何一条命中即阻断）

- [ ] Skill可调用高危操作但没有审批/限制
- [ ] 输入可注入并影响系统指令或工具执行
- [ ] 关键失败场景没有降级或回退路径
- [ ] 缺失最小可观测性（日志、trace、错误分类）
- [ ] Skill组合产生权限提升（超出各自权限并集）
- [ ] Skill间上下文传递导致跨租户/跨用户数据泄漏
- [ ] 链式调用故障传播无隔离（单Skill故障导致级联崩溃）

## 三、评审清单（高价值）

### 1) 设计质量
- [ ] 技能职责单一，边界清晰
- [ ] 输入输出可验证，格式稳定
- [ ] 失败行为被定义（返回错误而非静默失败）

### 2) 安全合规
- [ ] 最小权限 + allowlist
- [ ] 敏感信息不进入日志与提示词
- [ ] 数据访问遵循租户/用户隔离

### 3) Skill 组合与编排
- [ ] Skill间调用有显式权限声明，组合后权限不超出各自并集
- [ ] 上下文传递经过安全过滤，不含敏感信息泄漏
- [ ] 链式调用有独立超时与熔断，单Skill故障不级联
- [ ] 多Skill并发时资源争抢（token预算/上下文窗口）有优先级策略
- [ ] Skill激活/去激活无竞态条件
- [ ] 矛盾指令有冲突检测与解决策略

### 4) 可靠性
- [ ] 超时、重试和熔断策略存在
- [ ] 外部依赖失败时可降级
- [ ] 关键操作具备幂等和补偿

### 5) 运营可维护
- [ ] 指标可观测（成功率、时延、成本、错误率）
- [ ] 质量评估可复现（固定样本集/回归集）
- [ ] 版本变更有兼容策略

## 四、评分与发布决策（风险调整版）

> 本节替代原纯频率评分，采用风险调整评分。详细方法论参见 `14_nondeterministic_eval_guide.md` 第三节。

### 4.1 风险调整评分公式

```
Risk-Adjusted Score = 100 - (Failure_Rate × Severity_Weight)
```

### 4.2 严重度权重

| 失败类型 | 权重 | 示例 |
|---------|------|------|
| 灾难性 | 100 | 数据泄漏、安全绕过、组合权限提升 |
| 严重 | 50 | 功能不可用、错误决策、级联故障 |
| 中等 | 20 | 降级但可用、响应质量下降 |
| 轻微 | 5 | 格式问题、体验不佳 |

### 4.3 等级映射

- **A（90-100）**：可发布
- **B（80-89）**：修复P2后发布
- **C（70-79）**：修复P1后复审
- **D（<70）**：禁止发布，需重构

> 强制规则：任何 P0/P1 未关闭，不允许发布。
> 判定优先级：**阻断规则 > 发布评分**（先看阻断，再看分数）。
> 灾难性失败（权重100）发生次数 > 0 即直接 D 级，无论频率多低。

## 四点五、最小量化阈值（默认建议）

- Skill 成功率：建议 `>= 99%`（需附样本量与置信区间，参见 `14_nondeterministic_eval_guide.md`）
- P95 时延：建议在业务 SLO 内（默认可参考 `< 3s`）
- 高危工具误触发率：建议 `= 0`
- 回归集通过率：建议 `>= 95%`
- 单次发布后回滚率：建议 `< 5%`
- Skill组合权限提升路径数：必须 `= 0`
- 链式调用级联故障率：建议 `< 0.1%`

## 四点六、回归测试方法论

### 4.6.1 回归集构建

- **最小规模**：覆盖所有 P0/P1 场景 + 核心业务路径，不少于 50 个测试用例
- **用例来源**：60% 生产流量采样 + 20% 边界/异常场景 + 20% 历史缺陷回归
- **标注要求**：每个用例标注：输入、期望输出、可接受偏差范围、严重度等级

### 4.6.2 非确定性回归判定

- 相同测试集 + 相同配置（temperature 等）运行 ≥3 次
- 成功率取均值，允许 ±2% 统计波动（根据实际方差调整）
- 使用 Welch's t-test 对比新旧版本，p < 0.05 且 Cohen's d > 0.2 才判定为"有差异"
- 效应量判定：d < 0.2 等价/0.2-0.5 小差异/0.5-0.8 中等差异/>0.8 大差异

### 4.6.3 回归通过标准

| 指标 | 通过条件 |
|------|---------|
| 成功率 | 新版本 ≥ 旧版本 - 2%（统计波动容忍） |
| 灾难性失败数 | 新版本 = 0 |
| P95 时延 | 新版本 ≤ 旧版本 × 1.1 |
| Token 成本 | 新版本 ≤ 旧版本 × 1.05 |

## 四点七、30分钟快审流程

1. **5分钟**：扫阻断规则（第二节，含组合阻断项）
2. **5分钟**：检查关卡材料是否齐备（第一节）
3. **5分钟**：核查组合安全与回归结果（第三节第3项 + 四点六）
4. **5分钟**：核对量化阈值与风险调整评分（四点五 + 第四节）
5. **10分钟**：给发布结论与修复清单

## 五、Skill 组合审查详细方法

### 5.1 信息流分析

```
对每对可组合的 Skill (A, B):
  1. 枚举 A 的输出集合 output_A
  2. 枚举 B 的输入集合 input_B
  3. 计算 output_A ∩ input_B（信息流交叉点）
  4. 对每个交叉点检查：
     - A 的输出是否包含 B 不应获得的信息
     - A→B 的信息流是否创建了新的权限路径
     - 组合后权限是否超出 A 和 B 各自权限的并集
```

### 5.2 故障传播分析

```
对链式调用 A → B → C:
  1. 枚举每个 Skill 的故障模式
  2. 分析 A 故障对 B、C 的影响
  3. 分析 B 故障对 C 的影响
  4. 验证每个环节有独立熔断/降级
  5. 确认单点故障不会级联到整条链
```

### 5.3 资源争抢分析

```
对并发运行的多 Skill 集合:
  1. 枚举共享资源（token预算、上下文窗口、API配额）
  2. 分析资源耗尽场景
  3. 验证优先级策略存在且合理
  4. 验证低优先级 Skill 不会饿死高优先级 Skill
```

## 六、2026前沿建议（仅保留核心）

- 把评估与安全策略纳入 CI（Eval/Security as Gate）
- 建立"技能回归集"，每次变更自动对比质量与成本
- 增加"人工确认网关"覆盖高风险工具
- 统一事件模型，支持跨技能追踪与审计
- Skill 组合权限图自动生成与检测
- 非确定性回归测试纳入 CI 流水线
- 对抗性红队测试作为发布前必经关卡（参见 `15_adversarial_review_guide.md`）
