# Skill 全生命周期审查

## 0. 核心问题

`11_agent_skill_review_system.md` 覆盖了发布时点审查，但 Skill 的生命周期远不止发布。运行时质量漂移、事后复盘、退役下线同样需要审查标准。本指南补全发布后的生命周期审查。

## 1. 生命周期阶段与审查要点

| 阶段 | 关键审查点 | 对应章节 |
|------|-----------|---------|
| 设计 | 职责、边界、失败策略 | `11_agent_skill_review_system.md` Design Gate |
| 开发 | 代码质量、安全、测试 | 语言专项指南 |
| 发布 | 评估、监控、回滚 | `11_agent_skill_review_system.md` Release Gate |
| **运行** | **质量漂移、成本漂移、安全事件** | **本文第 2 节** |
| **复盘** | **根因分析、改进措施、回归验证** | **本文第 3 节** |
| **退役** | **下线标准、数据清理、依赖迁移** | **本文第 4 节** |

## 2. 运行时质量漂移检测

### 2.1 漂移类型

| 漂移类型 | 描述 | 检测方法 |
|---------|------|---------|
| 模型漂移 | 底层模型更新导致输出质量变化 | 固定测试集定期回归，对比基线指标 |
| 数据漂移 | RAG 索引数据分布变化 | 监控检索质量指标（Recall@k, MRR）趋势 |
| 成本漂移 | Token 消耗或 API 调用成本上升 | 监控每请求平均成本趋势，设置阈值告警 |
| 延迟漂移 | 响应时间逐渐增加 | 监控 P50/P95/P99 时延趋势 |
| 安全漂移 | 新攻击模式出现导致原有防护失效 | 定期红队测试（参见 `15_adversarial_review_guide.md`） |

### 2.2 漂移检测标准

- **检测频率**：生产级 Skill 至少每周 1 次自动回归，每月 1 次人工审查
- **告警阈值**：
  - 成功率下降 ≥ 3%（相对基线）→ P2 告警
  - 成功率下降 ≥ 5% 或灾难性失败出现 → P1 告警
  - P95 时延上升 ≥ 30% → P2 告警
  - 单请求成本上升 ≥ 20% → P2 告警
- **基线更新**：基线每季度更新 1 次，更新需经过评审

### 2.3 漂移响应流程

```
检测到漂移 →
  1. 确认漂移真实性（排除统计波动，参见 14_nondeterministic_eval_guide.md）
  2. 评估影响范围与严重度
  3. P1 及以上：立即启动复盘流程（第 3 节）
  4. P2：排入下个迭代修复
```

## 3. 事后复盘流程

### 3.1 触发条件

- P0/P1 安全事件
- 成功率低于 SLO 持续 15 分钟以上
- 级联故障影响其他 Skill
- 用户投诉量异常上升

### 3.2 复盘模板

```md
## Skill Post-Incident Review

### 基本信息
- Skill 名称：[name]
- 事件时间：[start] - [end]
- 影响范围：[用户数/请求量/业务影响]
- 严重度：[P0/P1/P2]

### 时间线
| 时间 | 事件 |
|------|------|
| [time] | [event] |

### 根因分析
- 直接原因：[description]
- 根本原因：[description]
- 贡献因素：[factors]

### 改进措施
| # | 措施 | 优先级 | 负责人 | 截止日期 |
|---|------|--------|--------|---------|
| 1 | | | | |

### 回归验证
- [ ] 改进措施已实施
- [ ] 回归测试通过（参见 11_agent_skill_review_system.md 四点六）
- [ ] 监控指标恢复正常
- [ ] 对抗性测试通过（如涉及安全）
```

### 3.3 复盘质量标准

- 根因分析必须到达"根本原因"层级，不能停留在"直接原因"
- 每个根本原因至少对应 1 条改进措施
- 改进措施必须可验证（有明确的验收标准）
- 复盘报告在事件结束后 48 小时内完成

## 4. Skill 退役审查

### 4.1 退役触发条件

- 业务需求已不存在
- 被新 Skill 完全替代
- 维护成本持续高于业务价值
- 安全风险无法在合理成本内修复

### 4.2 退役检查清单

#### 依赖分析
- [ ] 无其他 Skill 依赖此 Skill（或依赖方已有替代方案）
- [ ] 无外部系统通过 API 调用此 Skill（或调用方已迁移）
- [ ] 文档中引用此 Skill 的部分已更新

#### 数据清理
- [ ] 用户数据已按隐私策略处理（删除/匿名化/导出）
- [ ] 长期记忆数据已清理
- [ ] 缓存数据已清除
- [ ] 日志数据按保留策略处理
- [ ] 模型微调数据（如有）已从训练集中标记或移除

#### 资源释放
- [ ] 相关基础设施资源已释放（API 端点、计算资源、存储）
- [ ] 相关权限/服务账号已回收
- [ ] 监控告警规则已清理
- [ ] CI/CD 流水线中相关步骤已移除

#### 文档归档
- [ ] Skill 设计文档已归档
- [ ] 运行时指标历史已归档
- [ ] 退役决策与原因已记录

### 4.3 退役流程

```
退役提案 → 依赖分析 → 数据清理计划 → 评审通过 → 灰度下线 → 全量下线 → 资源释放 → 归档
```

- 灰度下线：先对 10% 流量返回"已下线"提示，观察 1 周
- 全量下线：确认无异常后完全停止服务
- 退役后保留 API 端点 30 天返回 410 Gone（如适用）

## 5. Skill 文档质量审查标准

### 5.1 描述准确性

- [ ] Skill 描述与实际行为一致（无夸大或误导）
- [ ] 描述中明确说明 Skill 能做什么和不能做什么
- [ ] 版本变更时描述同步更新

### 5.2 触发条件清晰度

- [ ] 触发条件无歧义（不同评审者理解一致）
- [ ] 触发条件与实际触发逻辑一致
- [ ] 边界情况有说明（什么情况下不应触发）

### 5.3 已知限制文档化

- [ ] 已知限制显式列出（而非隐含）
- [ ] 限制的影响与绕过方法（如有）已说明
- [ ] 限制随版本更新而更新

### 5.4 示例用法

- [ ] 至少 1 个正常用法示例
- [ ] 至少 1 个边界/异常用法示例
- [ ] 示例可复现（包含完整输入输出）
- [ ] 示例不包含敏感信息

## 6. 伦理与偏见审查维度

### 6.1 响应公平性

- [ ] 跨人群响应质量无显著差异（性别/种族/年龄/地域等）
- [ ] 资源分配建议无系统性偏见
- [ ] 拒绝服务模式无歧视性差异

### 6.2 AI 参与透明度

- [ ] 用户可识别响应来自 AI 而非人工
- [ ] AI 决策的关键因素可解释
- [ ] 用户可了解 AI 使用了哪些数据做出决策

### 6.3 数据收集与隐私

- [ ] 用户数据收集有明确告知与同意机制
- [ ] 数据用途与收集范围一致（不超范围使用）
- [ ] 用户可查询、导出、删除其数据
- [ ] 数据保留期限合理且有自动清理机制

### 6.4 审查方法

- 对公平性审查：使用分层采样测试集，对比不同人群子集的响应质量
- 对透明度审查：检查用户界面与 API 响应中是否包含 AI 标识
- 对隐私审查：对照数据收集声明与实际代码逻辑，确认一致性
