# P0-P3 分级矩阵（Severity Matrix）

> 本文档定义 `jdcloud-alert-intelligence` v0.1 的告警分级规则。
> 全部基于**规则引擎**，不依赖 ML/统计（ML 留给 v0.3）。
> 配套阅读：[`./playbook-classify.md`](./playbook-classify.md)（分级脚本与伪代码）。

## 1. 矩阵总览

### 1.1 三维判定

| 维度 | 取值 | 说明 |
|:--|:--|:--|
| **影响面 (Scope)** | S1 单资源 / S2 同服务 2-2 资源 / S3 同服务 ≥3 资源 / S4 跨服务 | S 由同 service_code 下受影响 resource_id 数量决定 |
| **持续时间 (Duration)** | D1 < 5min / D2 5-30min / D3 30min-2h / D4 > 2h | 从告警簇首次触发到末次触发的时长 |
| **频次 (Frequency)** | `low` < 3 次/小时 / `mid` 3-10 次/小时 / `high` > 10 次/小时 | **次/小时口径**；时间窗默认 24h |

**业务影响等级 (Business Impact)** 是独立维度，**先于 S/D/F 判定**，见 §7。

**频次计算公式**：`频次 = 触发次数 / 时间窗小时数`（次/小时）
- `low`: < 3 次/小时
- `mid`: 3-10 次/小时
- `high`: > 10 次/小时
- 时间窗默认 24h；用户可指定 1h / 6h / 24h / 7d

> 权威源：[`./core-concepts.md §1.4`](./core-concepts.md) "频次维度"小节。

### 1.2 4 级速查

| 级别 | 核心条件（满足任一）| 处置时限 |
|:--|:--|:--|
| **P0** | 核心业务中断 / S3-S4 / D3-D4 / 业务影响等级=核心 | ≤ 5 分钟响应 |
| **P1** | 单资源 D2 / 业务影响等级=重要 | ≤ 15 分钟响应 |
| **P2** | 单资源 D1 / 业务影响等级=一般 | ≤ 1 小时响应 |
| **P3** | 已知周期性 / 维护窗 / 自动恢复 / 业务影响等级=边缘 | 仅日报聚合 |

### 1.3 完整判定矩阵（20 单元）

下表为分级速查。S/D/F 的语义见 §1.1。空白单元表示该组合下"需结合业务影响等级二次判定"（见 §7）。

| 单元 | 业务影响=核心 | 业务影响=重要 | 业务影响=一般 | 业务影响=边缘 |
|:--|:--:|:--:|:--:|:--:|
| **S3-S4 / D3-D4** | P0 | P0 | P0 | P1 |
| **S3-S4 / D1-D2** | P0 | P1 | P1 | P2 |
| **S2 / D3-D4** | P0 | P1 | P1 | P2 |
| **S2 / D1-D2** | P1 | P1 | P2 | P3 |
| **S1 / D4** | P0 | P1 | P2 | P3 |
| **S1 / D3** | P1 | P1 | P2 | P3 |
| **S1 / D2** | P1 | P2 | P2 | P3 |
| **S1 / D1 / `high` 频次** | P1 | P2 | P3 | P3 |
| **S1 / D1 / `low`/`mid` 频次** | P2 | P2 | P3 | P3 |
| **维护窗内（任意 S/D/F）** | 降一档 | 降一档 | 降一档 | 降一档 |
| **自动恢复（末次触发 > 30min 无新触发）** | 降一档 | 降一档 | 降一档 | 降一档 |
| **高频抖动（S1/D1 + `high` 频次）** | 升一档 | 不变 | 不变 | 不变 |
| **周期性（历史 7d 同时段 ≥3 次）** | 降一档 | 降一档 | 降一档 | 降一档 |
| **资源属核心业务白名单** | P0 | P1 | P2 | P3 |
| **资源属边缘业务白名单** | 降一档 | 降一档 | P3 | P3 |

> 完整 20 单元的边界细则见 §3-§6 各级别详判。
> 抑制降档规则（命中即降，**不跨级降**）见 §8。

## 2. 业务影响等级定义

| 等级 | 判定方法 | 典型场景 |
|:--|:--|:--|
| **核心 (core)** | 资源标签 `business=core` 或属于订单/支付/登录/交易链路的 resource_id | 订单库主库、支付回调服务、登录认证服务 |
| **重要 (important)** | 资源标签 `business=important` 或属于报表/对账/通知 | 报表查询库、对账服务、短信通知服务 |
| **一般 (general)** | 资源标签 `business=general` 或未打标签且无业务画像 | 普通业务服务、内部工具、测试环境 |
| **边缘 (peripheral)** | 资源标签 `business=peripheral` 或资源 7d 内 QPS < 10 | 文档站点、静态页面、备份任务节点 |

> **权威源**：[`./core-concepts.md §1.4`](./core-concepts.md) "业务影响等级"小节。

**v0.1 业务画像识别方式：**

1. 优先读 JD Cloud 资源 tag（`describe-instances` 返回 `tags` 字段）
2. 次之读用户在 `jdcloud-skill-generator` 阶段录入的业务画像表
3. 最后回退到"未打标签 = `general`"

## 3. P0 详判（最高优先级）

### 3.1 P0 触发条件（满足任一即为 P0）

| 编号 | 条件 | 检测方法 |
|:--|:--|:--|
| P0-1 | **核心业务中断**：业务影响等级=核心 且 (S2-S4 或 D3-D4 或 `high` 频次) | 资源 tag 含 `business=core` 且矩阵落在 P0 区域 |
| P0-2 | **多资源同时异常**：同 service_code 下受影响 resource_id ≥ 3 | 聚合后 cluster.resources.length ≥ 3 |
| P0-3 | **跨服务故障**：≥ 2 个不同 service_code 同时段告警且任一为 vm/rds/lb | 检查同时段 cluster，service_code distinct count ≥ 2 |
| P0-4 | **超长持续**：单资源持续 > 30min 未恢复 | cluster.duration > 30min |
| P0-5 | **核心白名单命中**：资源在 `references/playbook-classify.md § 6` 的核心白名单内（如 `rds-mysql-*` 主库、`lb-*` 核心入口） | 资源 ID 前缀 + tag 双重匹配 |

### 3.2 P0 处置建议模板

```
🚨 P0 告警簇: <cluster.summary>
  - 首次触发: <first_trigger_time>
  - 持续时长: <duration> 分钟
  - 受影响资源: <resource_id_list>
  - 业务影响: <business_impact_label>
  - 下一跳建议:
    → 如果是 VM/Redis/LB 资源问题: jdcloud-vm-ops / jdcloud-redis-ops / jdcloud-clb-ops
    → 如果是 RDS 主库问题: jdcloud-rds-ops (主从切换/连接池)
    → 如果是告警规则误配: jdcloud-cloudmonitor-ops (调整阈值)
  - 响应时限: ≤ 5 分钟
```

### 3.3 P0 真实场景示例

| 场景 | 判定路径 | 原因 |
|:--|:--|:--|
| 订单库主库 CPU > 95% 持续 40min | P0-1 + P0-4 | 核心业务 + 超长持续 |
| 同一 LB 后端 3 台 ECS 同时 CPU 100% | P0-2 | 多资源同时异常 |
| 交易链路：支付服务 + 订单库同时段告警 | P0-3 | 跨服务故障 |
| 边缘备份任务 CPU 100% 持续 2h | **不是 P0** | 业务影响=边缘，降为 P1 |

## 4. P1 详判（高优先级）

### 4.1 P1 触发条件（满足任一即为 P1）

| 编号 | 条件 | 检测方法 |
|:--|:--|:--|
| P1-1 | **单资源持续 5-30min**：S1 + D2，且业务影响等级=核心或重要 | cluster.resources.length=1 且 5min < duration ≤ 30min |
| P1-2 | **重要业务中等持续**：业务影响等级=重要 + (S2 / D3) | 重要业务 + 中等影响 |
| P1-3 | **核心业务低频抖动**：业务影响等级=核心 + S1 + D1 + `high` 频次 | 核心业务 + 频繁抖动 |
| P1-4 | **资源属核心白名单 + 短持续**：白名单资源 + D1-D2 | 即使短暂也需关注 |
| P1-5 | **从 P0 降档**：满足 P0 条件但已自动恢复（30min 内无新触发） | 末次触发距今 > 30min |
| P1-6 | **从 P0 降档**：满足 P0 条件但命中维护窗 | 见 §8 抑制降档 |

### 4.2 P1 处置建议模板

```
⚠️  P1 告警簇: <cluster.summary>
  - 首次触发: <first_trigger_time>
  - 持续时长: <duration> 分钟
  - 受影响资源: <resource_id_list>
  - 业务影响: <business_impact_label>
  - 下一跳建议:
    → jdcloud-*-ops 排查资源健康
    → 如反复出现,考虑 jdcloud-cloudmonitor-ops 调整阈值
  - 响应时限: ≤ 15 分钟
```

### 4.3 P1 真实场景示例

| 场景 | 判定路径 | 原因 |
|:--|:--|:--|
| 单台应用服务器 CPU 90% 持续 15min | P1-1 | 单资源中等持续 |
| 报表库连接数 80% 持续 45min | P1-1 | 重要业务 + 中等持续 |
| 订单服务频繁 5xx 抖动 30 次/小时 | P1-3 | 核心业务 + 高频抖动 |
| 边缘 VM CPU 100% 持续 2h | **不是 P1** | 业务影响=边缘 → P3 |

## 5. P2 详判（中优先级）

### 5.1 P2 触发条件（满足任一即为 P2）

| 编号 | 条件 | 检测方法 |
|:--|:--|:--|
| P2-1 | **单资源短抖动**：S1 + D1（< 5min），业务影响=一般/边缘 | 单资源短持续 |
| P2-2 | **重要业务短持续**：业务影响=重要 + S1 + D1 | 重要业务偶发 |
| P2-3 | **同服务 2 资源短持续**：S2 + D1-D2 | 2 个资源同时短抖 |
| P2-4 | **从 P1 降档**：满足 P1 条件但已自动恢复 | 末次触发距今 > 30min |
| P2-5 | **从 P1 降档**：满足 P1 条件但命中周期性 | 历史 7d 同时段 ≥3 次 |

### 5.2 P2 处置建议模板

```
📋 P2 告警簇: <cluster.summary>
  - 业务影响: <business_impact_label>
  - 下一跳建议:
    → 工作时间排查，非紧急
    → 多次重复考虑 jdcloud-cloudmonitor-ops 调整阈值
  - 响应时限: ≤ 1 小时（工作时间）
```

### 5.3 P2 真实场景示例

| 场景 | 判定路径 | 原因 |
|:--|:--|:--|
| 普通应用服务器 CPU 95% 持续 3min 后恢复 | P2-1 | 单资源短抖动 |
| 报表库连接数 75% 持续 2min | P2-2 | 重要业务短持续 |
| 同 LB 后端 2 台 ECS CPU 90% 持续 4min | P2-3 | 2 资源短持续 |

## 6. P3 详判（低优先级 / 降档落点）

### 6.1 P3 触发条件（满足任一即为 P3）

| 编号 | 条件 | 检测方法 |
|:--|:--|:--|
| P3-1 | **已知周期性**：历史 7d 同时段（±30min）≥ 3 次 | 见 [`./suppression-rules.md`](./suppression-rules.md) § 3 |
| P3-2 | **维护窗内**：资源 tag 含 `maintenance_window=*` 或用户在分析时显式声明的维护期 | 资源 tag 匹配 |
| P3-3 | **已自动恢复**：告警簇末次触发距当前分析时间 > 30min | 时间差判定 |
| P3-4 | **业务影响=边缘**：资源属边缘业务白名单 | 资源 tag 或 7d QPS < 10 |
| P3-5 | **已知误报**：命中 `suppression-rules.md § 4` 的 5 类常见 case | 规则匹配 |
| P3-6 | **降档结果**：从 P2 降档后落入 P3 区域 | 抑制降档后位置 |

### 6.2 P3 处置建议

```
ℹ️  P3 告警簇: <cluster.summary>
  - 抑制原因: <suppression_reason>
  - 下一跳建议:
    → 无需立即响应
    → 计入日报/周报，作为长期优化信号
    → 如反复抑制，考虑 jdcloud-cloudmonitor-ops 调整阈值或加抑制规则
  - 响应时限: 仅日报聚合
```

### 6.3 P3 真实场景示例

| 场景 | 判定路径 | 原因 |
|:--|:--|:--|
| 每日 3:00 备份任务 CPU 100% 持续 10min | P3-1 | 已知周期性 |
| 已发布维护公告的 VM 在维护窗内重启 | P3-2 | 维护窗内 |
| 边缘文档站点 CPU 偶发 90% | P3-4 | 业务影响=边缘 |
| 镜像仓库拉取时网络抖动 | P3-5 | 已知误报 |

## 7. 边界 case 与误判纠正

### 7.1 5 个常见误判场景

| 场景 | 易错判定 | 正确判定 | 说明 |
|:--|:--|:--|:--|
| **5min 持续抖动算 P1 还是 P2?** | P1（"单资源持续 5-30min"） | P1 | D2 下限 5min 包含（闭区间），但需要业务影响 ≥ 重要；否则 P2 |
| **边缘资源持续 2h** | P0（"持续 > 30min"） | P3 | 业务影响等级=边缘优先于 D 维度，降档 |
| **核心业务偶发 1 次** | P0（"核心业务"） | P2 | 核心业务需要满足 S/D/F 至少一个，不是单凭标签 |
| **3 个资源同时抖动 1min** | P2（"短持续"） | P0 | S3 优先于 D1，多资源同时异常即 P0 |
| **1 个资源反复触发 100 次/小时** | P2（"单资源"） | P1 | `high` 频次升一档规则（核心业务）或 P1（其他） |

### 7.2 完整 10 个边界 case

| # | 场景 | 判定 | 关键依据 |
|:--|:--|:--|:--|
| 1 | 订单库主库 CPU 100% 持续 31min | **P0** | P0-1 + P0-4 |
| 2 | 订单库主库 CPU 100% 持续 29min | **P1** | P1-1（核心业务 + D2 边界） |
| 3 | 订单库主库 CPU 100% 持续 29min 已恢复 | **P2** | P2-4（从 P1 降档） |
| 4 | 边缘 VM CPU 100% 持续 5h | **P3** | 业务影响=边缘 + P3-4 |
| 5 | 普通应用 CPU 90% 持续 3min | **P2** | P2-1 |
| 6 | 普通应用 CPU 90% 持续 3min 后反复 50 次 | **P1** | `high` 频次升一档到 P1 |
| 7 | 同 LB 后端 3 台 ECS CPU 80% 持续 1min | **P0** | P0-2 |
| 8 | 同 LB 后端 3 台 ECS CPU 80% 持续 1min 但属边缘业务 | **P1** | 业务影响=边缘 → 降一档 |
| 9 | 核心 RDS 主从延迟 30s 持续 10min | **P1** | P1-1（核心 + D2） |
| 10 | 边缘 RDS 慢查询 100 次/min 持续 1h | **P3** | 业务影响=边缘 → P3 |

### 7.3 误判纠正原则

1. **业务影响等级 > S/D/F**：业务影响等级=边缘时，任意 S/D/F 组合最多 P1
2. **S 维度优先于 D 维度**：S3-S4 即使 D1 仍是 P0
3. **核心白名单加成**：白名单资源 + D1-D2 至少 P1
4. **自动恢复降档**：满足高级别条件但已恢复，降一档
5. **维护窗降档**：任意条件在维护窗内，降一档
6. **不能跨级降**：P0 不能直接降到 P3，必须先降到 P1，再由其他规则降到 P3

## 8. 抑制降档规则

执行顺序：维护窗 → 已知周期性 → 已知误报清单。

| 命中类型 | 降档 |
|---|---|
| 维护窗 | 降一档 |
| 已知周期性 | 降一档 |
| 已知误报清单（5 类）| 一律降为 P3 |

**约束**：

- P3 不再降
- 维护窗/周期性不跨级降（P0→P1，P1→P2，P2→P3）
- 已知误报清单跨级豁免（任意级别都降为 P3）
- 多个命中时按"最重抑制"原则：已知误报 > 已知周期性 > 维护窗

**权威源**：[`./core-concepts.md §1.4`](./core-concepts.md) "抑制降档 (两段式)"小节。

### 8.1 升档规则（v0.1 仅 1 条）

| 升档源 | 命中条件 | 升档效果 |
|:--|:--|:--|
| 高频抖动 | S1/D1 + `high` 频次 (>10 次/小时) + 业务影响=核心 | 升一档（P2 → P1）|

### 8.2 不做（v0.1 范围外）

- ❌ ML 异常检测升档（v0.3）
- ❌ 跨服务根因升档（v0.2 需 rca-engine）
- ❌ 动态阈值修正（v0.3）

## 9. v0.1 已知局限

1. **业务画像依赖 tag**：未打标签的资源默认"一般"，可能与实际不符
2. **周期性检测窗口固定 7d**：不支持用户自定义窗口
3. **频次窗口与时间窗强绑定**：时间窗缩短会减少 F 维度数值
4. **跨服务关联未实现**：P0-3 检测需要 v0.2 rca-engine
5. **不识别应用层告警**：仅识别云资源层告警（vm.cpu / rds.qps 等）
