# 资源盘点报告格式规范 — `jdcloud-routines-ops`

> **版本**: v1.0 (2026-06-10 定型)
> **来源**: `jdcloud-routines-ops/outputs/customers/丹东鹏飞/部署结构分析报告-20260610.md` 从 v1.0 → v1.3 的复盘经验
> **目标读者**: 编写资源盘点报告的 LLM Agent / 人工运维
> **强制执行**: 所有 `jdcloud-routines-ops` 产出的客户资源盘点报告必须通过本规范 GCL 评分

---

## 1. 为什么需要这份规范

### 1.1 真实案例：丹东鹏飞报告的三轮强化

| 版本 | 问题发现 | 强化内容 | 根因 |
|---|---|---|---|
| v1.0 | 用户反馈"子网只显示名字，没显示 ID" | 9 个章节全部升级为 **子网名 + 子网 ID** 双标识 | 规则意识缺位：没想到运维需要 ID 调 API |
| v1.1 | 用户反馈"续费清单只有数量，没有实例 ID" | §5.2 升级为 **13 个到期资源的完整实例 ID 清单** | 执行者视角缺位：没想到工单需要具体 ID |
| v1.2 | 用户指出"标签错误归因不严谨" | 修正措辞为"三方字段矛盾"，明确是**云上资源状态**而非 skill bug | 归因框架缺位：没区分"云上问题"vs"skill 问题" |

### 1.2 核心原则

> **报告不是为了"好看"，而是为了"可执行"**

每写一节，必须自问：**"读者拿到这节要做什么？"**

- 如果读者要**操作资源**（续费/变更/删除）→ 必须提供可直接使用的 **ID/API 参数**
- 如果读者要**了解现状**（拓扑/配置）→ 必须提供可追溯的 **双标识（Name + ID）**

## 2. 强制执行规范（P0/P1）

### 2.1 P0 规范（一票否决，必须修复）

#### R1：双标识规则（Name + ID）

**适用范围**：报告中凡涉及**子网/VPC/安全组/路由表/磁盘/快照**等可引用资源

**规范**：必须同时显示 **Name + ID**，缺一不可，格式统一为：
```
**子网名** / `subnet-id`
**VPC名** / `vpc-id`
**安全组名** / `sg-id`
```

**正例**：
```markdown
| 主机名 | **子网名 + ID** |
|---|---|
| 鹏飞-erp-prd-oracle | **鹏飞-ERP-DB** / `subnet-ue3wimfa0j` |
```

**反例**：
```markdown
| 主机名 | 子网 |  ❌ 缺 ID |
|---|---|
| 鹏飞-erp-prd-oracle | 鹏飞-ERP-DB |  ❌ 缺 ID |

| 主机名 | 子网 |  ❌ 缺 Name |
|---|---|
| 鹏飞-erp-prd-oracle | subnet-ue3wimfa0j |  ❌ 缺 Name |
```

**理由**：
- Name 是给人看的（运维一眼识别业务分段）
- ID 是给脚本/API 用的（调 `jdc` 或 SDK 时必须用 ID）
- 两者互补，缺一不可

---

#### R2：操作性清单必须含具体实例 ID

**适用范围**：**续费/变更/删除/启动/停止**等可操作清单

**规范**：每行必须含**具体实例 ID**（如 `i-9i22fdurlq`、`redis-jq8j5c6slnfd`），不得仅以"12 台 VM"聚合

**正例**：
```markdown
| # | 资源名 | **实例 ID** | 续费 Skill | 到期日 |
|---|---|---|---|---|
| 1 | 鹏飞-erp-prd-oracle | **i-9i22fdurlq** | jdcloud-vm-ops | 2026-06-15 |
| 2 | 鹏飞-erp-prd-pg | **i-2qz6biqitv** | jdcloud-vm-ops | 2026-06-15 |
```

**反例**：
```markdown
⚠️ 12 台 VM 即将到期，请尽快续费。  ❌ 不可执行，需要 ID 才能调 jdc/SDK
```

**理由**：
- 工单系统、jdc CLI、SDK 调用都需要具体 ID
- "12 台 VM"这种聚合描述需要二次查询，降低执行效率
- 具体 ID 也是可追溯性的基础（审计时能精准定位）

---

### 2.2 P1 规范（建议修复，可讨论豁免）

#### R3：归因责任矩阵

**规范**：风险项描述必须区分**"云上资源客观状态"** vs **"Skill 自身问题"**

**判定流程**：
```
发现数据矛盾（如 name/vpc/tag 不一致）
    ↓
是 skill 采集/写入导致的吗？
    ├─ 是 → 描述为"Skill 采集错误"或"Skill 写入错误"
    └─ 否（skill 是 read-only，只是镜像云上状态）
        ↓
        描述为"云上资源三方字段矛盾（name/vpc/tag 不一致）"
        并加脚注：此为云上资源客观状态，非 skill 自身问题
```

**正例**：
```markdown
| 现象 | 资源 | 说明 |
|---|---|---|
| **三方字段不一致** | `subnet-ru6zazmhqu` | `vpcId=vpc-kxzah06nqx`(狗道宠物) + `subnetName=狗道宠物-SLB-Internet` + `tags[客户]=丹东鹏飞` 三方矛盾。**此为云上资源客观状态，非 `jdcloud-aiops-cruise` skill 自身问题**（skill 是 read-only 嗅探，不打标签不改字段） |
```

**反例**：
```markdown
| 现象 | 资源 | 说明 |
|---|---|---|
| **标签错误** | `subnet-ru6zazmhqu` | 被打成 `客户=丹东鹏飞`（实际属狗道宠物 VPC）  ❌ 未区分是云上问题还是 skill 问题 |
```

---

#### R4：执行者视角

**规范**：每节必须自问：**"读者拿到这节要做什么？"**

**检查清单**：
- [ ] 如果读者要操作资源，是否提供了可直接使用的 ID/API 参数？
- [ ] 如果读者要了解现状，是否提供了可追溯的标识（Name + ID）？
- [ ] 是否避免了"二次查询"（如只给数量不给 ID）？
- [ ] 是否提供了可直接粘贴执行的 YAML/命令行片段（如续费 Checklist）？

---

#### R5：章节自检清单

**规范**：提交前必须对照《资源盘点报告自检清单》（§3）逐项勾选

**违规处理**：
- 无自检记录 → `Spec Compliance = 0.5`（建议修复）
- 自检通过但事后发现违规 → 回溯优化自检清单本身

## 4. 典型章节格式模板

### 4.1 业务系统划分章节（§3）

```markdown
### 3.X [系统名](N 台 VM)

| 主机名 | **实例 ID** | IP | 规格 | OS | 用途 | **子网名 + ID** | 计费到期 |
|---|---|---|---|---|---|---|---|
| 鹏飞-erp-prd-oracle | **i-9i22fdurlq** | 172.21.14.19 | c.n3.2xlarge (8C/16G) | CentOS 7.9 | Oracle DB | **鹏飞-ERP-DB** / `subnet-ue3wimfa0j` | 2026-06-15 |
| 鹏飞-erp-prd-pg | **i-2qz6biqitv** | 172.21.14.20 | c.n3.2xlarge (8C/16G) | CentOS 7.9 | PostgreSQL | **鹏飞-ERP-DB** / `subnet-ue3wimfa0j` | 2026-06-15 |

> **子网分布**：
> - **鹏飞-ERP-DB** / `subnet-ue3wimfa0j` (172.21.14.16/28)：2 台（Oracle + PG）
> - **鹏飞-应用生产** / `subnet-cpfablqwyk` (172.21.14.32/27)：3 台（app-1/2/3）
```

**要点**：
- 实例 ID 用 `**粗体**` 突出
- 子网列统一格式：`子网名 / subnet-id`
- 子网分布总结也要带 ID

| 2 | 鹏飞-erp-prd-pg | **i-2qz6biqitv** | 172.21.14.20 | c.n3.2xlarge | **鹏飞-ERP-DB** / `subnet-ue3wimfa0j` | CentOS 7.9 | jdcloud-vm-ops | 2026-06-15 |

### 4.3 风险清单章节（§8）

```markdown
| # | 风险项 | 详情 | 建议行动 |
|---|---|---|---|
| **R1** | **批量到期 5 天** | 12 VM(**i-9i22fdurlq** 等)+ 1 Redis(**redis-jq8j5c6slnfd**)共 13 个 prepaid 资源，2026-06-15 统一到期。**详见 §5.2 完整实例 ID 清单** | 立即联系客户续费，提交 `jdcloud-vm-ops` (12 IDs) + `jdcloud-redis-ops` (1 ID) 续费工单 |
```

- 建议行动必须具体到 skill 名称和资源 ID

## 5. GCL 评分集成

### 5.1 在 `references/rubric.md` 中的位置

本规范的 R1-R5 对应 rubric 的 **Dimension 5: Spec Compliance** 的子项。

**更新后的 Spec Compliance 评分细则**：

| 子项 | 权重 | 评分标准 |
|---|---|---|
| R1 双标识规则 | 0.3 | 所有子网/VPC/SG 同时显示 Name + ID。缺一项 → 0.5；缺两项 → 0 |
| R2 操作性清单 | 0.3 | 续费/变更/删除清单每行含具体实例 ID。有聚合描述 → 0.5；无 ID → 0 |
| R3 归因责任 | 0.2 | 数据矛盾类风险区分云上问题 vs skill 问题。未区分 → 0.5；错误归因 → 0 |
| R4 执行者视角 | 0.1 | 提供可直接粘贴执行的 YAML/命令行。需二次查询 → 0.5 |
| R5 自检清单 | 0.1 | 报告末尾含自检清单签名。无签名 → 0.5 |

**总 Spec Compliance 分数** = 各子项加权平均
